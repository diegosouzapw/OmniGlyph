# การมีส่วนร่วมกับ OmniGlyph

ขอบคุณที่สนใจ! โปรเจกต์นี้มีกฎวัฒนธรรมที่ไม่สามารถต่อรองได้สองข้อ —
นี่คือเหตุผลที่ทำให้ทุกตัวเลขใน README เชื่อถือได้

## กฎข้อที่ 1 — TDD แบบเข้มงวด

โค้ดสำหรับใช้งานจริงทุกบรรทัดเกิดจากการทดสอบที่ล้มเหลวก่อน:

1. เขียนการทดสอบและ **ดูมันล้มเหลวด้วยเหตุผลที่ถูกต้อง**
2. เขียนโค้ดขั้นต่ำสุดเพื่อให้ผ่าน
3. ปรับปรุงโครงสร้างโดยยังคงสถานะเขียว

มาตรฐานเต็มคือ: `pnpm run typecheck && pnpm test && pnpm run build` — ทั้งสาม
คำสั่งเสมอ (การตรวจลิงก์เอกสารและตัวป้องกัน rebrand รันอยู่ภายใน `pnpm test`
ผ่าน `tests/docs-integrity.test.ts`)

## กฎข้อที่ 2 — การวัดผลก่อนการกล่าวอ้าง

การเปลี่ยนแปลงใดๆ ต่อรูปทรง, atlas, สูตรคิดค่าใช้จ่าย หรือขอบเขตโมเดล จะไม่ถูก merge
โดยไม่มีตัวเลขที่วัดได้จริง repository นี้ถูกสร้างขึ้นบนวินัยนี้:

- ต้นทุนค่าใช้จ่าย → พิสูจน์ด้วย `benchmarks/billing-sweep/` (`count_tokens`
  ใช้ฟรี; ค่าคลาดเคลื่อนที่คาดหวัง: ศูนย์)
- ความสามารถในการอ่าน → พิสูจน์ด้วย `benchmarks/density-frontier/` (n≥30 ต่อกลุ่ม
  การให้คะแนนแบบกำหนดแน่นอน หลักฐาน JSONL ถูก commit ไว้ใน `benchmarks/*/results/`)
- เกณฑ์ยอมรับสำหรับการเปลี่ยนค่าเริ่มต้นในการใช้งานจริง: gist == เท่ากับ baseline ข้อความ
  **และ** ไม่มี error แบบ silent exact-string เลย **และ** มีการประหยัดที่เป็นบวก

สมมติฐานที่ยังไม่มีตัวเลขจะไปอยู่ที่ `docs/ROADMAP.md` ในฐานะสมมติฐาน — ไม่เคยเข้าไปใน
README ในฐานะข้อเท็จจริง มีไอเดีย "ที่ดูชัดเจน" สองอย่างที่ถูกหักล้างด้วยข้อมูลแล้ว
(หน้าความละเอียดสูง, atlas แบบ anti-aliased); กระบวนการนี้ใช้ได้ผลจริง

## การตั้งค่า

```bash
pnpm install
pnpm test              # ชุดทดสอบทั้งหมด, ~40–90 วินาที
pnpm run dev:node      # พร็อกซีในเครื่องแบบ watch mode
```

Node ≥18 (CI ทดสอบบน 20/22/24), pnpm 10 (ล็อกเวอร์ชันด้วย `packageManager` ใน
package.json)

## โครงสร้าง

| โฟลเดอร์ | กฎ |
|---|---|
| `src/core/` | ไม่ขึ้นกับรันไทม์ (ใช้ Web API เท่านั้น — รันได้ทั้งบน Node และ Workers) |
| `src/node.ts` / `src/worker.ts` | ทำหน้าที่เชื่อมต่อโฮสต์เท่านั้น |
| `benchmarks/` | ชุดทดสอบที่รันซ้ำได้; ผลลัพธ์ JSONL คือหลักฐาน ต้อง commit |
| `docs/` | benchmarks/ (ตัวเลข), architecture/ (แผนผัง), ROADMAP (สมมติฐาน), ops/ (OmniRoute) |

## Commit และ PR

- ใช้ conventional commits (`feat:`, `fix:`, `perf:`, `docs:`, `refactor:`,
  `test:`, `chore:`) ส่วนเนื้อหาต้องอธิบาย *เหตุผล* พร้อมตัวเลขที่เกี่ยวข้อง
- PR ขนาดเล็ก มีจุดโฟกัสชัดเจน; การเปลี่ยนแปลงพฤติกรรมต้องมาพร้อมการทดสอบที่ยึดผลไว้ และ
  ถ้าเกี่ยวข้อง ต้องมี benchmark ที่สนับสนุนการเปลี่ยนแปลงนั้น
- ห้ามเขียนบล็อก `cache_control` ของไคลเอนต์ใหม่ ห้ามเพิ่ม dependency ที่รันไทม์
  โดยไม่ปรึกษาก่อน (แกนหลักตั้งใจให้มี dependency น้อยที่สุด) ห้ามใช้
  `Math.random`/timestamp ในเส้นทางการเรนเดอร์ (ความแน่นอนของผลลัพธ์เป็น invariant สำคัญ
  ที่ทดสอบด้วยการเปรียบเทียบไบต์ต่อไบต์)

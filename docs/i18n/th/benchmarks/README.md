## วิธีที่การประหยัดทำงาน (ในภาพเดียว)

ผู้ให้บริการคิดค่าใช้จ่าย **ข้อความตามจำนวนโทเคน** แต่คิดค่าใช้จ่าย
**ภาพตามขนาดของมัน** — ไม่ใช่ตามปริมาณข้อความที่บรรจุอยู่ข้างใน หน้า
มาตรฐานหนึ่งหน้ามีต้นทุนคงที่ไม่ว่าข้อความจะหนาแน่นแค่ไหน:

```
                         ┌─────────────────────────────────────────────┐
28,080 characters   →    │  one 1568 × 728 PNG page   =  1,460 tokens   │
of dense context         └─────────────────────────────────────────────┘
                                          the SAME 1,460 whether the page
                                          holds 200 chars or 28,080
```

บริบทเดียวกัน คิดค่าใช้จ่ายสองแบบ:

```
as dense TEXT    ██████████████████████████████████████████████  ~14,040 tokens
as ONE IMAGE     █████                                              1,460 tokens
                                                                    ▼
                                                              ~10× fewer tokens
```

เหตุผลที่ภาพชนะ — จำนวนตัวอักษรที่บรรจุต่อโทเคน:

```
image (dense page)  ███████████████████████████████████████  19.2 chars/token
dense text          ████                                       ~2  chars/token
```

OmniGlyph จะสลับใช้วิธีนี้ก็ต่อเมื่อคณิตศาสตร์แบบเป๊ะบอกว่าคุ้มเท่านั้น
และเฉพาะกับโมเดลที่พิสูจน์แล้วว่าอ่านหน้าได้ ชุดทดสอบทั้งสองด้านล่างพิสูจน์
แต่ละครึ่งหนึ่ง

## 1. `billing-sweep/` — ภาพหนึ่งภาพมีต้นทุนจริงเท่าไร?

การตรวจสอบ `count_tokens` แบบฟรีเทียบกับ API ของ Anthropic ที่ใช้งานจริง
เปรียบเทียบสูตรที่เลิกใช้แล้ว `w·h/750` กับโมเดลแพตช์ 28px ปัจจุบัน
ใน 11 รูปทรงการตรวจสอบ บน 2 โมเดล × 2 ระดับความละเอียด

**ผลลัพธ์ (2026-07-05): โมเดลแพตช์เข้ากันได้พอดีด้วยค่าคลาดเคลื่อน ศูนย์ ในทุก
การตรวจสอบ** — คิดค่าใช้จ่าย = `⌈w/28⌉ × ⌈h/28⌉` หลังปรับขนาดตามระดับ
บวกค่าคงที่ +3/+4 โทเคนต่อบล็อกภาพ หน้าที่ใช้งานจริง
(1568×728) มีต้นทุนเท่ากับ 1,460 โทเคนพอดี และบรรจุ 28,080 ตัวอักษร ≈
**19.2 ตัวอักษร/โทเคน** เทียบกับ ~2 ตัวอักษร/โทเคนของข้อความหนาแน่น

```
the page as the patch grid the API actually bills:

  ⌈1568 / 28⌉ = 56 patches wide
  ⌈ 728 / 28⌉ = 26 patches tall
  ───────────────────────────────
  56 × 26  = 1,456  + 4 per-block  =  1,460 tokens   (flat, WYSIWYG)
```

```bash
node benchmarks/billing-sweep/run.mjs --dry-run          # predictions only, $0
ANTHROPIC_API_KEY=... node benchmarks/billing-sweep/run.mjs   # live sweep, still $0 (count_tokens is free)
```

## 2. `density-frontier/` — โมเดลอ่านได้จริงหรือไม่?

ต้นทุน (ออฟไลน์, แบบเป๊ะ) × ความแม่นยำในการอ่าน (ใช้งานจริง) ข้ามคอนฟิก
การเรนเดอร์ รูปทรงหน้า atlas กลิฟ และผู้ให้บริการ corpus ปลูก
needle แบบ string ที่เป๊ะ (id เป็น hex, camelCase, ลำดับตัวเลข) บวก
**ตัวหลอกแบบ near-miss ที่สร้างจากคู่ความสับสนของกลิฟ
ที่วัดได้จริง** — เพื่อให้การกุเรื่องแบบเงียบถูกตรวจจับได้ ไม่ใช่แค่ถูก
นับว่าผิด การให้คะแนนแบบกำหนดแน่นอน (ไม่มี LLM เป็นกรรมการ):
`correct` / `abstained` (`ILEGIVEL` แบบซื่อสัตย์) / `silent_wrong` / `no_answer`

**ผลลัพธ์หลัก** (n=30 ต่อกลุ่ม):

| กลุ่ม | อ่านได้เป๊ะ | หมายเหตุ |
|---|---:|---|
| Fable 5 · หน้ามาตรฐาน · atlas 1-bit (ใช้งานจริง) | **30/30** | ไม่มี error ไม่มีการกุเรื่อง |
| Fable 5 · หน้ามาตรฐาน · atlas AA (ค่าเริ่มต้นเก่า) | 25/30 | งดตอบอย่างซื่อสัตย์ 5 ครั้ง — เหตุผลที่การใช้งานจริงเปลี่ยนเป็น 1-bit |
| Fable 5 · หน้าความละเอียดสูง 1928² | 1–2/30 | คิดค่าใช้จ่ายแพงกว่า 3.3 เท่า แต่ encoder ถูก resample — กับดักด้านค่าใช้จ่าย ไม่ได้เปิดใช้งาน |
| Opus 4.8 · กลิฟ 10×16 | 23–26/30 | โหมดปลอดภัยแบบเลือกเปิดใช้ |
| GPT-5.5 · แถบ 768px (ทั้งสอง atlas) | 0/60 | + โทเคนผลลัพธ์ขยาย ~40 เท่า เทียบกับกลุ่มควบคุมข้อความของตัวเอง (30/30, 62 โทเคน) |
| Gemini 2.5-flash (บางส่วน, โควตา) | 0/26 | กุเรื่องแทนที่จะงดตอบ |

ความแม่นยำในการอ่านแบบดูปราดเดียวเข้าใจ — นี่ **คือ** เกทแบบ fail-closed
ของโมเดล วาดเป็นภาพ:

```
Fable 5 · 1-bit page (prod)   ██████████████████████████████  30/30  ✅ approved
Opus 4.8 · 10×16 (safe mode)  ████████████████████████░░░░░░  ~24/30 ⚠️ opt-in
Fable 5 · high-res 1928²      █░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  ~2/30  🚫 billing trap
GPT-5.5 · 768px strip         ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  0/60   ⛔ blocked
Gemini 2.5-flash              ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  0/26   ⛔ confabulates
```

มีเพียงกลุ่ม ✅ เท่านั้นที่ถูกนำไปใช้งานจริง สิ่งใดที่อ่านได้ไม่ดีจะถูกบล็อก
*พร้อมหลักฐาน* และการให้คะแนนแบบสามทางหมายความว่าโมเดลที่เดาผิด
(`silent_wrong`) ถูกมองว่าแย่กว่าโมเดลที่งดตอบอย่างซื่อสัตย์ (`ILEGIVEL`)

สามช่องทางการส่งข้อมูล: API โดยตรง (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`/`GEMINI_API_KEY`),
OpenRouter (`OPENROUTER_API_KEY`) และ `--via-cli` (การสมัครสมาชิก Claude
Code — $0) ข้อควรระวังที่เรียนรู้มาแบบยากลำบาก: ตัวกลาง (OpenRouter,
เครื่องมือ Read ของ CLI) จะปรับขนาดภาพใหญ่ มีเพียงผลลัพธ์จาก
API โดยตรงเท่านั้นที่เชื่อถือได้สำหรับความสามารถในการอ่าน

```bash
pnpm exec tsx benchmarks/density-frontier/run.ts --dry-run                    # cost table, $0
pnpm exec tsx benchmarks/density-frontier/run.ts --via-cli --sections 30     # via subscription, $0
ANTHROPIC_API_KEY=... pnpm exec tsx benchmarks/density-frontier/run.ts --configs anthropic-std-5x8-1bit
```

`tests/anthropic-vision.test.ts`, `tests/gemini-profiles.test.ts`,

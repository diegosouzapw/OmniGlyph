# 📊 The numbers — measured, not estimated

| ตัวชี้วัด | ผลลัพธ์ | หลักฐาน |
|---|---|---|
| การลดค่าใช้จ่ายแบบครบวงจร | **59–70%** | ข้อมูลการใช้งานจริง 13,709 คำขอ |
| โทเคนต่อบล็อกที่ถูกแปลง | **น้อยลง 10 เท่า** (28,080 ตัวอักษร: 14,040 → 1,460 โทเคน) | [billing sweep](../../../benchmarks/billing-sweep/README.md) |
| ความแม่นยำของสูตรคิดค่าใช้จ่าย | ค่าคลาดเคลื่อน **ศูนย์** จาก 22 การตรวจสอบ `count_tokens` 2 โมเดล × 2 ระดับ | `benchmarks/billing-sweep/results/` |
| ความแม่นยำในการอ่านแบบเป๊ะ, ค่าคอนฟิกที่ใช้งานจริง | **30/30 (100%)** บน Claude Fable 5 | [density frontier](../../../benchmarks/density-frontier/README.md) |
| การกุเรื่องแบบเงียบใน ~300 การทดสอบอ่าน | **0** — ทุกครั้งที่อ่านพลาด จะงดตอบเป็น `ILEGIVEL` | `benchmarks/density-frontier/results/` |

**ตารางคะแนนโมเดล** (อ่านเรนเดอร์ความหนาแน่นสูงได้หรือไม่? n=30 ต่อกลุ่ม การให้คะแนนแบบกำหนดแน่นอน):

| โมเดล | การอ่าน | คำตัดสิน |
|---|---|---|
| Claude **Fable 5** | **100%** เป๊ะ | ✅ เป้าหมายสำหรับใช้งานจริง |
| Claude Opus 4.8 | 77–87% ที่ขนาดกลิฟ 4 เท่า | ⚠️ โหมดปลอดภัยแบบเลือกเปิดใช้ (การประหยัดลดเหลือ ~2 เท่า) |
| GPT-5.5 | 0/60 — และตอบขยายคำตอบออกไป ~40 เท่าเมื่อพยายามอ่าน | ❌ ถูกบล็อกโดยเกท พร้อมหลักฐาน |

ข้อได้เปรียบนี้ **เจาะจงกับ Fable ในตอนนี้** — encoder การมองเห็นตัวอื่นยังอ่านกลิฟความหนาแน่นสูงไม่ได้ [ชุดทดสอบ benchmark](../../../benchmarks/README.md) จะทดสอบโมเดลใหม่ทุกตัวได้ในคำสั่งเดียว

# 🤔 ทำไมต้อง OmniGlyph?

ทุกเซสชันเอเจนต์ที่ทำงานยาวนานจะแบกน้ำหนักตายตัวเดิมซ้ำๆ ทุกคำขอ: system prompt, เอกสารเครื่องมือ และประวัติเก่า — ถูกคิดค่าใช้จ่ายต่อโทเคนซ้ำทุกรอบ OmniGlyph เป็น **พร็อกซีในเครื่อง** ที่เขียนส่วนที่ใหญ่เหล่านั้นใหม่เป็นหน้า PNG ความหนาแน่นสูง *ก่อนที่จะออกจากเครื่องของคุณ*:

- **คำนวณค่าใช้จ่ายแบบเป๊ะ ไม่ใช่ heuristic** — คำนวณสูตรโทเคนภาพจริงของผู้ให้บริการ (วัดได้ค่าคลาดเคลื่อนเป็นศูนย์) และแปลงเฉพาะเมื่อคุ้มค่าจริง
- **ออกแบบแบบ fail-closed** — โมเดลที่อ่านเรนเดอร์ความหนาแน่นสูงไม่ได้จะถูกบล็อกโดยเกท พร้อมหลักฐานจาก benchmark ไม่มีการสูญเสียคุณภาพแบบเงียบ
- **เป็นส่วนตัวและทำงานในเครื่องก่อน** — การเขียนใหม่เกิดขึ้นที่ `127.0.0.1`; ไม่มีอะไรถูกส่งออกไปที่อื่นเพิ่มเติม
- **ทำซ้ำผลได้** — ทุกตัวเลขข้างต้นมีหลักฐานอยู่ใน `benchmarks/*/results/` และรันซ้ำได้ในคำสั่งเดียว

# ⚡ เริ่มต้นอย่างรวดเร็ว

```bash
npx omniglyph                                     # พร็อกซีที่ 127.0.0.1:47821
ANTHROPIC_BASE_URL=http://127.0.0.1:47821 claude  # ชี้ Claude Code ไปที่พร็อกซีนี้
```

![เริ่มต้นอย่างรวดเร็ว: เริ่มพร็อกซี ตรวจสอบแดชบอร์ด แล้วชี้ Claude Code ไปที่มัน](../../../docs/assets/demo-quickstart.gif)

ใช้งานได้ทั้งสองแบบ:
- **API key** (จ่ายตามโทเคน): ค่าใช้จ่ายของคุณลด 59–70% แบบครบวงจร
- **เซสชันแบบสมัครสมาชิก**: คุณจะไม่จ่ายน้อยลง แต่ขีดจำกัดการใช้งานถูกนับเป็นโทเคน — ดังนั้นขีดจำกัดของคุณจะยืดออกไป **~2–3 เท่า**

แดชบอร์ดที่ <http://127.0.0.1:47821/>: โทเคนที่ประหยัดได้ ทุกการแปลงข้อความ→ภาพแบบเทียบเคียงกัน สวิตช์ปิดฉุกเฉิน ชิปแสดงโมเดลแบบเรียลไทม์ การตอบกลับสตรีมตามปกติ — มีเพียง *คำขอ* เท่านั้นที่ถูกบีบอัด ไม่ใช่ผลลัพธ์จากโมเดล

# ⚙️ วิธีการทำงาน

```
บล็อกคำขอขนาดใหญ่ ──► เกทตรวจสอบความคุ้มค่า ──► reflow + เรนเดอร์ (atlas 1-bit 5×8)
                       (คำนวณค่าใช้จ่ายแบบเป๊ะ)     ──► หน้า PNG 1568×728 ──► นำกลับเข้าที่เดิม เป็นมิตรกับแคช
```

- **คำนวณค่าใช้จ่ายแบบเป๊ะก่อนแปลง**: Anthropic คิดค่าใช้จ่าย `⌈w/28⌉ × ⌈h/28⌉ + 4` โทเคนต่อภาพ (แพตช์ขนาด 28 พิกเซล — วัดได้ค่าคลาดเคลื่อนเป็นศูนย์) หนึ่งหน้าเต็มบรรจุ 28,080 ตัวอักษรในราคา 1,460 โทเคน ≈ **19 ตัวอักษร/โทเคน** เทียบกับ ~2 ตัวอักษร/โทเคนของข้อความหนาแน่น เกทจะแปลงเฉพาะเมื่อคำนวณแล้วคุ้มค่า
- **สิ่งที่ถูกแปลง**: system prompt แบบคงที่ + เอกสารเครื่องมือ, ประวัติเก่าที่ถูกยุบรวม, ผลลัพธ์เครื่องมือขนาดใหญ่
- **สิ่งที่ไม่ถูกแปลงเลย**: ข้อความของคุณ, การสนทนาล่าสุด, ผลลัพธ์จากโมเดล, ข้อความที่เบาบาง, ค่าที่ต้องเป๊ะทุกไบต์ (hash/ID เดินทางเป็นข้อความควบคู่ไปด้วย) และโมเดลใดก็ตามที่สอบตก benchmark การอ่าน

# 📚 ใช้งานแบบไลบรารี (ไม่ผ่านพร็อกซี)

ทุกสิ่งที่พร็อกซีทำต่อคำขอหนึ่งครั้ง ก็มี API ที่มีเอกสารประกอบและนำเข้าใช้งานได้เช่นกัน:

```ts
import { renderTextToImages, transformAnthropicMessages } from "omniglyph";

// เรนเดอร์ข้อความใดๆ ให้เป็นหน้า PNG แบบ 1-bit ความหนาแน่นสูง
const { pages } = await renderTextToImages(bigToolOutput, { reflow: true });
// pages[i].png: Uint8Array · pages[i].width × pages[i].height

// หรือรันการแปลงคำขอแบบเต็มรูปแบบด้วยตัวเอง — เกท, สูตรคำนวณค่าใช้จ่าย ครบทุกอย่าง
const { body, applied, reason } = await transformAnthropicMessages({
  body: requestBytes,           // raw body ของ /v1/messages เป็น JSON
  model: "claude-fable-5",
});
```

`options.keepSharp(block)` ตรึงบล็อกให้เป็นข้อความ; `options.emitRecoverable` คืนค่าต้นฉบับของบล็อกที่ถูกแปลงเป็นภาพ สูตรคำนวณค่าใช้จ่ายแบบเป๊ะก็มีให้ใช้ที่รากของแพ็กเกจเช่นกัน (`anthropicImageTokens`, `resolveAnthropicVisionTier`, `openAIVisionTokens`) — นี่คือสิ่งที่ [OmniRoute](https://github.com/diegosouzapw/OmniRoute) นำไปใช้ รันไทม์เป็น Pure-JS ล้วน (Node และ edge/Workers) พื้นผิว API แบบเต็ม: `src/core/index.ts`

# 🧭 The honest part

- **มันเป็นแบบ lossy** การดึงข้อมูลแบบเป๊ะทุกไบต์จากภาพนั้นไม่น่าเชื่อถือโดยธรรมชาติ มาตรการที่นำมาใช้แล้ว: ตัวระบุที่ต้องเป๊ะเดินทางเป็นข้อความควบคู่กับภาพ และค่าคอนฟิกที่ใช้งานจริงที่วัดผลแล้วให้ผล **การกุเรื่องแบบเงียบเป็นศูนย์** — การอ่านที่ล้มเหลวจะงดตอบ
- **มีเพียง Fable 5 เท่านั้นที่ได้รับการอนุมัติในตอนนี้** พร้อมหลักฐาน GPT-5.5 และ Gemini 2.5-flash วัดผลแล้วว่าอ่านเรนเดอร์ความหนาแน่นสูงไม่ได้; Opus 4.8 ต้องการกลิฟขนาดใหญ่กว่า 4 เท่า เกทบังคับใช้กฎนี้
- **เราพบและหลีกเลี่ยงกับดักด้านค่าใช้จ่าย**: ระดับภาพความละเอียดสูงคิดค่าใช้จ่ายแพงกว่า 3.3 เท่าต่อหน้า แต่ encoder การมองเห็นไม่ได้รับความละเอียดที่เพิ่มขึ้นจริง — หน้าที่ใหญ่กว่ากลับอ่านได้ *แย่กว่า* วัดผลแล้ว บันทึกไว้ใน [docs/benchmarks/BENCHMARKS.md](../../../docs/benchmarks/BENCHMARKS.md) และไม่ได้เปิดใช้งาน
- ราคาเปลี่ยนแปลงได้; ตัวชี้วัดที่ยั่งยืนคือการลดโทเคน ซึ่งพร็อกซีบันทึกไว้ต่อคำขอเทียบกับค่าตรงข้ามที่วัดฟรีด้วย `count_tokens`

# 🧠 คำถามที่พบบ่อย

**ตัวเลข 59–70% เป็นแบบครบวงจร หรือเฉพาะคำขอที่ถูกแตะต้อง?**
แบบครบวงจร — ทั้งบิล เครื่องมือบีบอัดส่วนใหญ่รายงานการประหยัดเฉพาะส่วนที่ถูกแตะต้องเท่านั้น ซึ่งทำให้ตัวเลขดูดีเกินจริง ตัวหารของเราคือ *ทุก* คำขอ: คำขอเล็กๆ ที่เกทปล่อยผ่านโดยไม่แตะต้องอย่างถูกต้อง การเขียนและอ่านแคชทั้งหมด และโทเคนผลลัพธ์ทั้งหมด (ซึ่งพร็อกซีไม่เคยบีบอัด) ตัวเลขเฉพาะส่วนที่ถูกบีบอัดจะสูงกว่าและถูกอ้างอิงแยกต่างหาก ไม่เคยใช้เป็นตัวเลขหลัก

**วัดการประหยัดได้อย่างไร?**
วัดทั้งสองด้านของคำขอเดียวกัน ในช่วงเวลาเดียวกัน สำหรับทุก POST ไปยัง `/v1/messages` พร็อกซีจะยิงการตรวจสอบ `count_tokens` แบบฟรีบน body ต้นฉบับที่ยังไม่ถูกบีบอัด (ค่าตรงข้ามเชิงสมมติ) ควบคู่ไปกับการส่งต่อจริง และอ่านบล็อกการใช้งานที่ถูกคิดค่าใช้จ่ายจริงจากผู้ให้บริการออกจากคำตอบ — ทั้งสองค่าถูกบันทึกอยู่ในแถวเหตุการณ์เดียวกัน ราคาแคชถูกใช้เหมือนกันทั้งสองด้าน ดังนั้นส่วนลดจากแคชจะหักล้างกันและไม่สามารถถูกนับซ้ำเป็น "การประหยัด" ได้ สูตรอยู่ใน `src/core/baseline.ts`; คุณสามารถคำนวณย้อนกลับได้จากบันทึกเหตุการณ์ของคุณเอง

**ทำไมการอ่านพลาดถึงกลายเป็นการกุเรื่อง แทนที่จะเป็นข้อผิดพลาดในการอ่าน?**
เพราะการมองเห็นของโมเดลไม่ใช่ OCR: หน้าภาพจะกลายเป็น patch embeddings ไม่ใช่ตัวอักษรที่แยกจากกันชัดเจน จึงไม่มีค่าความมั่นใจต่อกลิฟแต่ละตัวให้ล้มเหลวอย่างชัดเจนได้ — เมื่อพิกเซลไม่สามารถกำหนดกลิฟได้แน่ชัด language prior ของโมเดลจะเติมช่องว่างด้วยสิ่งที่ดูสมเหตุสมผล กลไกนี้เองคือเหตุผลที่ OmniGlyph ออกแบบแบบ fail-closed กับเรื่องนี้: ค่าที่ต้องเป๊ะทุกไบต์จะเดินทางเป็นข้อความควบคู่กับภาพเสมอ โมเดลที่อ่านผิดจะถูกบล็อกโดยเกท และค่าคอนฟิกที่ใช้งานจริงที่วัดผลแล้วให้ผลการกุเรื่องแบบเงียบเป็น **ศูนย์** ใน ~300 การทดสอบอ่าน — การอ่านที่ล้มเหลวจะงดตอบ

**แล้วงานที่ต้องเป๊ะทุกไบต์ล่ะ (hash, ID, ความลับ)?**
การสนทนาล่าสุดและตัวระบุที่ต้องเป๊ะจะยังคงเป็นข้อความโดยการออกแบบ สำหรับงานที่ต้องเป๊ะทุกไบต์*ทั้งหมด* ให้ส่งงานเหล่านั้นไปยังโมเดลที่ไม่อยู่ใน allowlist (เช่น subagent บนโมเดล Claude ตัวอื่น) — สิ่งใดก็ตามที่อยู่นอก allowlist จะผ่านไปโดยไม่ถูกแตะต้องและเหมือนต้นฉบับทุกไบต์

**DeepSeek-OCR ไม่ได้พิสูจน์แล้วหรือว่าวิธีนี้ใช้ได้จริง?**
มันพิสูจน์ว่า *ช่องทาง* นี้ใช้งานได้จริง — ด้วยคู่ encoder/decoder ที่ถูกฝึกมาเพื่องานนี้โดยเฉพาะ ความกังขานี้มาจากยุคที่ยังไม่มีโมเดลใช้งานจริงมาตรฐานตัวใดอ่านเรนเดอร์ความหนาแน่นสูงได้เลย — สถานการณ์นั้นเปลี่ยนไปแล้ว และ[ตารางคะแนนโมเดล](../../../README.md#-the-numbers--measured-not-estimated) ด้านบนแสดงให้เห็นชัดเจนว่าใครอ่านมันได้ในวันนี้ พร้อมหลักฐาน [ชุดทดสอบ benchmark](../../../benchmarks/README.md) จะทดสอบโมเดลใหม่ทุกตัวได้ในคำสั่งเดียว — เกทเดินตามข้อมูล ไม่ใช่ตามกระแส

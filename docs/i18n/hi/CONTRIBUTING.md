# OmniGlyph में योगदान देना

आपकी रुचि के लिए धन्यवाद! इस प्रोजेक्ट के दो गैर-परक्राम्य संस्कृति नियम हैं —
यही कारण है कि README में हर आँकड़े पर भरोसा किया जा सकता है।

## नियम 1 — सख़्त TDD

सारा प्रोडक्शन कोड पहले असफल हुए एक टेस्ट से जन्म लेता है:

1. टेस्ट लिखें और **इसे सही कारण से असफल होते देखें**।
2. इसे पास कराने के लिए न्यूनतम कोड लिखें।
3. हरा (green) बनाए रखते हुए रीफ़ैक्टर करें।

पूरी बार यह है: `pnpm run typecheck && pnpm test && pnpm run build` — तीनों,
हमेशा (docs link-lint और rebrand गार्ड `tests/docs-integrity.test.ts` के
ज़रिए `pnpm test` के भीतर चलते हैं)।

## नियम 2 — दावों से पहले मापन

ज्यामिति, एटलस, बिलिंग फ़ॉर्मूला, या मॉडल दायरे में कोई भी बदलाव बिना मापे
गए आँकड़े के नहीं आता। यह रिपॉज़िटरी इसी अनुशासन के इर्द-गिर्द बनाई गई है:

- बिलिंग लागत → `benchmarks/billing-sweep/` से साबित करें (`count_tokens`
  मुफ़्त है; अपेक्षित अवशिष्ट: शून्य)।
- पठनीयता → `benchmarks/density-frontier/` से साबित करें (n≥30 प्रति भुजा,
  निर्धारक स्कोरिंग, `benchmarks/*/results/` में कमिट किए गए JSONL रसीदें)।
- प्रोडक्शन डिफ़ॉल्ट बदलने की स्वीकृति सीमा: gist == टेक्स्ट बेसलाइन
  **और** शून्य मौन सटीक-स्ट्रिंग त्रुटियाँ **और** सकारात्मक बचत।

बिना आँकड़ों वाली परिकल्पनाएँ `docs/ROADMAP.md` में परिकल्पना के रूप में
जाती हैं — README में तथ्यों के रूप में कभी नहीं। दो "स्पष्ट" विचारों को
पहले ही आँकड़ों से खारिज किया जा चुका है (हाई-रेस पेज, एंटी-अलियास्ड
एटलस); यह प्रक्रिया काम करती है।

## सेटअप

```bash
pnpm install
pnpm test              # full suite, ~40–90s
pnpm run dev:node      # local proxy in watch mode
```

Node ≥18 (CI 20/22/24 पर टेस्ट करता है), pnpm 10 (package.json में
`packageManager` द्वारा पिन किया गया)।

## संरचना

| फ़ोल्डर | नियम |
|---|---|
| `src/core/` | रनटाइम-अज्ञेय (केवल Web API — Node और Workers दोनों पर चलता है) |
| `src/node.ts` / `src/worker.ts` | केवल होस्ट प्लंबिंग |
| `benchmarks/` | दोबारा चलाने योग्य हार्नेस; JSONL परिणाम रसीदें हैं, कमिट किए गए |
| `docs/` | benchmarks/ (आँकड़े), architecture/ (मानचित्र), ROADMAP (परिकल्पनाएँ), ops/ (OmniRoute) |

## कमिट और PR

- परंपरागत कमिट (`feat:`, `fix:`, `perf:`, `docs:`, `refactor:`,
  `test:`, `chore:`), बॉडी में प्रासंगिक आँकड़ों के साथ *क्यों* की व्याख्या।
- छोटे, केंद्रित PR; व्यवहार परिवर्तनों के साथ वह टेस्ट आता है जो उन्हें
  पिन करता है और, जहाँ लागू हो, वह बेंचमार्क जो उन्हें उचित ठहराता है।
- क्लाइंट के `cache_control` ब्लॉक फिर से न लिखें, बिना चर्चा के रनटाइम
  डिपेंडेंसी न जोड़ें (कोर जानबूझकर डिपेंडेंसी-हल्का है), रेंडर पाथ में
  `Math.random`/टाइमस्टैंप का उपयोग न करें (नियतिवाद एक कठोर अपरिवर्तनीय
  है, जिसे बाइट-पहचान द्वारा परीक्षित किया जाता है)।

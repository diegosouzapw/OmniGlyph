## 1. Anthropicの課金（直接count_tokens、$0、11ジオメトリ×2モデル）

確認された式: ティアごとのリサイズ後に`tokens = ceil(w/28) × ceil(h/28)`、
**+3/ブロック（Fable 5）/ +4/ブロック（Sonnet 4.5）** — 全行で残差ZERO。

| プローブ | 寸法 | Fable 5（high-res） | Sonnet 4.5（standard） |
|---|---|---:|---:|
| doc anchor | 1092×1092 | 1524 | 1525 |
| doc anchor | 1000×1000 | 1299 | 1300 |
| standard page | 1568×728 | 1459 | 1460 |
| **large page** | **1928×1928** | **4764（WYSIWYG!）** | 1525（resample） |
| hi-res ceiling | 1960×1960 | 4764（clamp） | 1525 |
| hi-res long edge | 2576×1204 | 3959 | 1516 |
| tall strip | 768×1932 | 1935 | 1292（resample） |
| 2×1092²（multi） | — | 3048（= 2×1524） | 3049 |
| 21×280²（>20 imgs） | — | 2163 | 2164 |
| 20×280²+2100² | — | 6824（2100²→downscale、count_tokensではREJECTされない） | 3585 |

導出された決定（実装済み）: 厳密なパッチ単位のゲート; モデルごとのティア
（Fable/Mythos/Opus 4.8/4.7/Sonnet 5 = high-res）; `cols` 313→312。

## 2. 読み取り精度（density-frontier、hex/camelCase/digitのneedle + distractor）

## 補遺 2026-07-06（2）— direct API経由のGPT-5.5バッテリー: H7クローズ（失敗）

| アーム | verbatim | gist | 出力/回答 |

GPT-5.5は5×8グリフを読めず（0/60; gistすら成立しない）、解読を試みる過程で
完了トークンを約40倍膨張させる（質問あたり2.4-2.7kの推論トークン）—
プロンプトの節約分が出力に食い潰される。完璧なテキスト対照群は、コーパス/質問が
妥当であることを証明している。5.5のオプトイン扱いを確認・定量化; gpt-5.6
（デフォルト）はアクセス権がなくテスト不能のまま。今後（H12）: GPTのゲートは
プロンプトトークンだけでなく出力の膨張もモデル化する必要がある。

## 補遺 2026-07-06（3）— Gemini 2.5-flash（部分的: 無料枠のクォータが実行途中で枯渇）

クォータが尽きる前に通過した約26件の画像回答のうち: **正解0、棄権1、
作話約25件** — しかもグリフの混同ではなく、ランダムな数字である
（`indexLedgerInd → 0040375615`）、つまりテストされた密度（ネイティブtile
42文字/トークンとMEDIUM固定）ではエンコーダがほとんど何も見えておらず、
2.5-flashは棄権する代わりに**作話する**（ILEGIVELの指示を無視する）。
テキスト対照群: 通過分では3/3。出力の膨張なし（6-28トークン/回答）。

予備的な兆候: H5/H6は2.5-flashについてはNO寄りで、失敗モードはGPTより
**悪い**（棄権ではなく無自覚な作話）— Geminiにはプロキシ側の追加の安全策が
必要になるだろう。今後の課題: 有償クォータで、または別の日に再実行すること、
そしてgemini-2.5-proをテストすること（flashはファミリー中最も弱い読み手）。
native-tileページは依然として文書化された比率としては最良（42.3文字/トークン）;
疑わしいのは可読性のほうである。

コストに関する注記: 部分的なページ（コーパスの最後のページ）はtileレジームの下で
コストが悪化する（高さが低い → クロップ単位が小さい → タイル数が増える）—
最後のページを高さ1152pxにパディングすることは、Geminiを導入する場合の
必須の最適化である。

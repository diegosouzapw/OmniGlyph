# مشارکت در OmniGlyph

از علاقهٔ شما سپاسگزاریم! این پروژه دو قانون فرهنگی غیرقابل‌مذاکره دارد —
همان دلیلی که هر عدد در README قابل اعتماد است.

## قانون ۱ — TDD سخت‌گیرانه

هر کد تولیدی از دل تستی زاده می‌شود که ابتدا شکست خورده است:

1. تست را بنویسید و **ببینید که به دلیل درست شکست می‌خورد**.
2. حداقل چیزی که آن را پاس می‌کند بنویسید.
3. در حالی که سبز می‌ماند، بازآرایی (refactor) کنید.

معیار کامل این است: `pnpm run typecheck && pnpm test && pnpm run build` — هر
سه، همیشه (بررسی لینک اسناد و گارد rebrand درون `pnpm test` از طریق
`tests/docs-integrity.test.ts` اجرا می‌شوند).

## قانون ۲ — اندازه‌گیری پیش از ادعا

هیچ تغییری در هندسه، اطلس، فرمول صورت‌حساب یا دامنهٔ مدل بدون یک عدد
اندازه‌گیری‌شده وارد نمی‌شود. این مخزن حول این انضباط ساخته شده است:

- هزینهٔ صورت‌حساب → آن را با `benchmarks/billing-sweep/` اثبات کنید
  (`count_tokens` رایگان است؛ باقی‌مانده مورد انتظار: صفر).
- خوانایی → آن را با `benchmarks/density-frontier/` اثبات کنید (n≥30 برای
  هر بازو، امتیازدهی قطعی، رسیدهای JSONL متعهد‌شده به `benchmarks/*/results/`).
- معیار پذیرش برای تغییر یک پیش‌فرض تولیدی: gist == خط پایهٔ متنی **و**
  صفر خطای رشته‌ای دقیق بی‌صدا **و** صرفه‌جویی مثبت.

فرضیه‌های بدون عدد به `docs/ROADMAP.md` به‌عنوان فرضیه می‌روند — هرگز به
README به‌عنوان واقعیت. دو ایدهٔ «بدیهی» تاکنون با داده رد شده‌اند (صفحهٔ
وضوح‌بالا، اطلس آنتی‌آلیاس‌شده)؛ این فرایند کار می‌کند.

## راه‌اندازی

```bash
pnpm install
pnpm test              # مجموعهٔ کامل، ~40 تا 90 ثانیه
pnpm run dev:node      # پراکسی محلی در حالت watch
```

Node ≥18 (CI روی نسخه‌های 20/22/24 تست می‌کند)، pnpm 10 (پین‌شده توسط
`packageManager` در package.json).

## ساختار

| پوشه | قانون |
|---|---|
| `src/core/` | مستقل از زمان اجرا (فقط Web APIs — روی Node و Workers اجرا می‌شود) |
| `src/node.ts` / `src/worker.ts` | فقط سیم‌کشی هاست (host plumbing) |
| `benchmarks/` | مجموعه‌های قابل اجرای مجدد؛ نتایج JSONL رسید هستند، متعهد‌شده |
| `docs/` | benchmarks/ (اعداد)، architecture/ (نقشه)، ROADMAP (فرضیه‌ها)، ops/ (OmniRoute) |

## کامیت‌ها و PRها

- کامیت‌های استاندارد (conventional commits) (`feat:`، `fix:`، `perf:`،
  `docs:`، `refactor:`، `test:`، `chore:`)، بدنه‌ای که *چرا* را با اعداد
  مرتبط توضیح دهد.
- PRهای کوچک و متمرکز؛ تغییرات رفتاری با تستی که آن‌ها را پین می‌کند و،
  در صورت لزوم، بنچمارکی که آن‌ها را توجیه می‌کند همراه می‌شوند.
- بلاک‌های `cache_control` سمت کلاینت را بازنویسی نکنید، بدون بحث
  وابستگی زمان اجرای جدید اضافه نکنید (هستهٔ پروژه عمداً سبک از وابستگی
  است)، در مسیرهای رندر از `Math.random`/برچسب‌های زمانی استفاده نکنید
  (قطعیت (determinism) یک ثابت سخت است، با تطابق بایتی تست می‌شود).

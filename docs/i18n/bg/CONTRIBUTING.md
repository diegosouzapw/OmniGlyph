# Принос към OmniGlyph

Благодарим за интереса! Този проект има две неотменими културни правила —
те са причината всяко число в README-то да е достойно за доверие.

## Правило 1 — Строг TDD

Целият производствен код се ражда от тест, който първо се проваля:

1. Напишете теста и **го гледайте как се проваля по правилната причина**.
2. Напишете минимума, за да мине.
3. Рефакторирайте, докато остава зелен.

Пълната летва е: `pnpm run typecheck && pnpm test && pnpm run build` — и трите,
винаги (линтването на връзки в документацията и пазителят на ребрандинга се изпълняват вътре в `pnpm test`
чрез `tests/docs-integrity.test.ts`).

## Правило 2 — Измерване преди твърдения

Никаква промяна в геометрията, атласа, формулата за таксуване или обхвата на моделите не се приема без
измерено число. Хранилището е изградено около тази дисциплина:

- Цена на таксуване → доказва се с `benchmarks/billing-sweep/` (`count_tokens` е
  безплатен; очакван остатък: нула).
- Четимост → доказва се с `benchmarks/density-frontier/` (n≥30 на рамо,
  детерминистично оценяване, JSONL доказателства, качени в `benchmarks/*/results/`).
- Летвата за приемане при промяна на производствена настройка по подразбиране: gist == текстова база
  **И** нула мълчаливи грешки при точни низове **И** положителни спестявания.

Хипотези без числа отиват в `docs/ROADMAP.md` като хипотези — никога в
README-то като факти. Две "очевидни" идеи вече бяха опровергани с данни
(страницата с висока резолюция, атласът с anti-aliasing); процесът работи.

## Настройка

```bash
pnpm install
pnpm test              # пълен набор от тестове, ~40–90s
pnpm run dev:node      # локален прокси в режим на наблюдение
```

Node ≥18 (CI тества 20/22/24), pnpm 10 (закован от `packageManager` в
package.json).

## Структура

| папка | правило |
|---|---|
| `src/core/` | независим от средата на изпълнение (само Web API — работи на Node и Workers) |
| `src/node.ts` / `src/worker.ts` | само инфраструктура за хоста |
| `benchmarks/` | повторно изпълними harness-ове; JSONL резултатите са доказателства, качени в хранилището |
| `docs/` | benchmarks/ (числа), architecture/ (карта), ROADMAP (хипотези), ops/ (OmniRoute) |

## Комити и PR-ове

- Conventional commits (`feat:`, `fix:`, `perf:`, `docs:`, `refactor:`,
  `test:`, `chore:`), с тяло, обясняващо *защо* заедно със съответните числа.
- Малки, фокусирани PR-ове; промените в поведението идват със теста, който ги закотвя, и,
  когато е приложимо, бенчмарка, който ги оправдава.
- Не пренаписвайте клиентските блокове `cache_control`, не добавяйте нови зависимости за изпълнение
  без обсъждане (ядрото умишлено е с малко зависимости), не използвайте
  `Math.random`/времеви маркери в пътищата за рендиране (детерминизмът е твърд инвариант,
  тестван чрез идентичност на байтовете).

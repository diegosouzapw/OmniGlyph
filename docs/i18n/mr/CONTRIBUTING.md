# OmniGlyph मध्ये योगदान

तुमच्या स्वारस्याबद्दल धन्यवाद! या प्रकल्पाचे दोन बिनतडजोड सांस्कृतिक नियम आहेत —
तेच कारण आहे की README मधील प्रत्येक आकड्यावर विश्वास ठेवता येतो.

## नियम 1 — कठोर TDD

सर्व उत्पादन कोड आधी अयशस्वी झालेल्या चाचणीतून जन्माला येतो:

1. चाचणी लिहा आणि **ती योग्य कारणासाठी अयशस्वी होताना पहा**.
2. ती पास होण्यासाठी किमान आवश्यक कोड लिहा.
3. हिरवे राहत रिफॅक्टर करा.

पूर्ण बार आहे: `pnpm run typecheck && pnpm test && pnpm run build` — तिन्ही,
नेहमी (docs link-lint आणि rebrand guard `pnpm test` मध्येच
`tests/docs-integrity.test.ts` द्वारे चालतात).

## नियम 2 — दाव्यांआधी मोजमाप

भूमिती, अॅटलास, बिलिंग सूत्र, किंवा मॉडेल व्याप्तीत कोणताही बदल मोजलेल्या
आकड्याशिवाय स्वीकारला जात नाही. हा रिपॉझिटरी याच शिस्तीभोवती बांधला आहे:

- बिलिंग खर्च → `benchmarks/billing-sweep/` सह सिद्ध करा (`count_tokens`
  मोफत आहे; अपेक्षित अवशेष: शून्य).
- वाचनीयता → `benchmarks/density-frontier/` सह सिद्ध करा (प्रति आर्म n≥30,
  निश्चयात्मक स्कोअरिंग, `benchmarks/*/results/` मध्ये कमिट केलेल्या JSONL
  पावत्या).
- उत्पादन डीफॉल्ट बदलण्यासाठीचा स्वीकृती बार: gist == मजकूर बेसलाइन
  **आणि** शून्य मूक अचूक-स्ट्रिंग त्रुटी **आणि** सकारात्मक बचत.

आकड्यांशिवाय गृहीतके README मध्ये तथ्य म्हणून नव्हे तर `docs/ROADMAP.md`
मध्ये गृहीतके म्हणून जातात. दोन "स्पष्ट" कल्पना आधीच डेटाद्वारे खोडून काढल्या
गेल्या आहेत (उच्च-रिझोल्यूशन पान, अँटी-अलायझ्ड अॅटलास); ही प्रक्रिया कार्य करते.

## सेटअप

```bash
pnpm install
pnpm test              # full suite, ~40–90s
pnpm run dev:node      # local proxy in watch mode
```

Node ≥18 (CI 20/22/24 तपासते), pnpm 10 (package.json मधील `packageManager`
द्वारे पिन केलेले).

## संरचना

| फोल्डर | नियम |
|---|---|
| `src/core/` | रनटाइम-अज्ञेय (फक्त Web APIs — Node आणि Workers दोन्हीवर चालतो) |
| `src/node.ts` / `src/worker.ts` | फक्त होस्ट प्लंबिंग |
| `benchmarks/` | पुन्हा-चालवण्यायोग्य हार्नेस; JSONL निकाल पावत्या आहेत, कमिट केलेल्या |
| `docs/` | benchmarks/ (आकडे), architecture/ (नकाशा), ROADMAP (गृहीतके), ops/ (OmniRoute) |

## कमिट्स आणि PRs

- कन्व्हेन्शनल कमिट्स (`feat:`, `fix:`, `perf:`, `docs:`, `refactor:`,
  `test:`, `chore:`), संबंधित आकड्यांसह *का* हे स्पष्ट करणारा बॉडी मजकूर.
- लहान, केंद्रित PRs; वर्तन बदल त्यांना पिन करणाऱ्या चाचणीसह आणि, लागू असल्यास,
  त्यांचे समर्थन करणाऱ्या बेंचमार्कसह येतात.
- क्लायंटचे `cache_control` ब्लॉक्स पुन्हा लिहू नका, चर्चेशिवाय रनटाइम
  डिपेंडन्सी जोडू नका (कोर हेतुपुरस्सर डिपेंडन्सी-हलका आहे), रेंडर पथांमध्ये
  `Math.random`/टाइमस्टॅम्प्स वापरू नका (निश्चयात्मकता हा एक कठोर invariant
  आहे, बाइट-आयडेंटिटीद्वारे चाचणी केलेला).

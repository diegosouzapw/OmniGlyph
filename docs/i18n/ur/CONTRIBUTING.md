# OmniGlyph میں تعاون کرنا

آپ کی دلچسپی کا شکریہ! اس پروجیکٹ کے دو ناقابلِ مذاکرات ثقافتی اصول ہیں —
یہی وجہ ہے کہ README کا ہر عدد قابلِ اعتماد ہے۔

## اصول 1 — سخت TDD

تمام پروڈکشن کوڈ ایک ایسے ٹیسٹ سے جنم لیتا ہے جو پہلے ناکام ہوا:

1. ٹیسٹ لکھیں اور **دیکھیں کہ وہ صحیح وجہ سے ناکام ہوتا ہے**۔
2. اسے پاس کرنے کے لیے کم از کم لکھیں۔
3. سبز رہتے ہوئے دوبارہ ساخت بندی کریں۔

مکمل معیار یہ ہے: `pnpm run typecheck && pnpm test && pnpm run build` — تینوں،
ہمیشہ (docs link-lint اور ری برانڈ گارڈ `pnpm test` کے اندر ہی
`tests/docs-integrity.test.ts` کے ذریعے چلتے ہیں)۔

## اصول 2 — دعوے سے پہلے پیمائش

جیومیٹری، ایٹلس، بلنگ فارمولے، یا ماڈل کے دائرہ کار میں کوئی تبدیلی بغیر
پیمائش شدہ عدد کے شامل نہیں ہوتی۔ ریپوزٹری اسی ضبط کے گرد تعمیر کی گئی ہے:

- بلنگ لاگت → اسے `benchmarks/billing-sweep/` سے ثابت کریں (`count_tokens`
  مفت ہے؛ متوقع بقیہ: صفر)۔
- پڑھنے کی صلاحیت → اسے `benchmarks/density-frontier/` سے ثابت کریں (فی بازو
  n≥30، متعین اسکورنگ، `benchmarks/*/results/` میں کمٹ شدہ JSONL رسیدیں)۔
- پروڈکشن ڈیفالٹ تبدیل کرنے کا قبولیت کا معیار: gist == متن بیس لائن
  **اور** صفر خاموش عین-اسٹرنگ غلطیاں **اور** مثبت بچت۔

بغیر اعداد کے مفروضات `docs/ROADMAP.md` میں بطور مفروضات جاتے ہیں — کبھی
حقائق کے طور پر README میں نہیں۔ دو "واضح" خیالات پہلے ہی ڈیٹا سے مسترد
کیے جا چکے ہیں (ہائی-ریز صفحہ، اینٹی-ایلیازڈ ایٹلس)؛ یہ عمل کام کرتا ہے۔

## سیٹ اپ

```bash
pnpm install
pnpm test              # مکمل سویٹ، ~40–90 سیکنڈ
pnpm run dev:node      # واچ موڈ میں لوکل پراکسی
```

Node ≥18 (CI ٹیسٹ 20/22/24)، pnpm 10 (package.json میں `packageManager` کے
ذریعے پن شدہ)۔

## ساخت

| فولڈر | اصول |
|---|---|
| `src/core/` | رن ٹائم-غیر جانبدار (صرف Web APIs — Node اور Workers دونوں پر چلتا ہے) |
| `src/node.ts` / `src/worker.ts` | صرف ہوسٹ پلمبنگ |
| `benchmarks/` | دوبارہ چلانے کے قابل ہارنسز؛ JSONL نتائج رسیدیں ہیں، کمٹ شدہ |
| `docs/` | benchmarks/ (اعداد)، architecture/ (نقشہ)، ROADMAP (مفروضات)، ops/ (OmniRoute) |

## کمٹس اور PRs

- روایتی کمٹس (`feat:`, `fix:`, `perf:`, `docs:`, `refactor:`,
  `test:`, `chore:`)، متعلقہ اعداد کے ساتھ *کیوں* کی وضاحت کرنے والا باڈی۔
- چھوٹے، مرکوز PRs؛ رویے کی تبدیلیاں اس ٹیسٹ کے ساتھ آتی ہیں جو انہیں پن کرتا
  ہے اور، جہاں لاگو ہو، اس بینچ مارک کے ساتھ جو انہیں جواز فراہم کرتا ہے۔
- کلائنٹ کے `cache_control` بلاکس کو دوبارہ نہ لکھیں، بغیر بحث کے رن ٹائم
  ڈیپینڈنسیز شامل نہ کریں (کور بالارادہ ڈیپینڈنسی-ہلکا ہے)، رینڈر پاتھس میں
  `Math.random`/ٹائم اسٹیمپس استعمال نہ کریں (تعین پذیری ایک سخت انویریئنٹ ہے،
  بائٹ-شناخت کے ذریعے ٹیسٹ شدہ)۔
